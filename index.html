Quiz 1 — Sample Question
A CloudOps engineer manages EC2 instances behind an Application Load Balancer. The team needs a single notification sent to an SNS topic only when average CPU utilization exceeds 80% AND the ALB healthy host count drops below 2 at the same time. Which approach meets this requirement?
  • Create a single metric alarm that uses metric math to combine both metric values into one expression
  • Create two separate metric alarms, each publishing to the same SNS topic
  • ✓ Create two metric alarms and combine them using a composite alarm with AND logic
  • Create an EventBridge rule that evaluates both CloudWatch metrics and triggers SNS
Show Explanation

- Why correct: A CloudWatch composite alarm lets you combine the states of multiple metric alarms using Boolean logic (AND/OR). By creating one metric alarm for CPU > 80% and another for healthy host count < 2, then combining them with AND in a composite alarm, the team receives a single SNS notification only when both conditions are simultaneously true. - Why A is wrong: Composite alarms evaluate alarm states, not raw metric values. Metric math can combine metrics into a single expression, but it cannot enforce 'both thresholds breached at the same time' logic across unrelated metrics the way composite alarms can. - Why B is wrong: Two separate alarms would each fire independently. The team would receive two separate notifications and could not enforce the requirement that both conditions must be true at the same time before alerting. - Why D is wrong: EventBridge can react to CloudWatch alarm state changes, but it does not evaluate metric thresholds or enforce AND logic between alarm conditions. It is a routing service, not a metric evaluation engine.

Quiz 2 — Sample Question
A company's Auto Scaling group of web servers is experiencing intermittent high CPU utilization that degrades response times. The operations team wants to be alerted when average CPU exceeds 80% for 5 consecutive minutes AND automatically add capacity. Which combination of AWS services achieves this with the LEAST operational overhead?
  • Create a CloudWatch alarm on average CPUUtilization with a 5-minute period and 1 evaluation period, configure the alarm to trigger an Auto Scaling scaling policy
  • Use a Lambda function on a 1-minute schedule to call DescribeInstances, calculate average CPU, and call SetDesiredCapacity when threshold is breached
  • ✓ Create a CloudWatch alarm on average CPUUtilization with a 1-minute period and 5 evaluation periods, configure the alarm to trigger an Auto Scaling scaling policy
  • Configure CloudTrail to detect high CPU events and use EventBridge to invoke a Lambda function that modifies the Auto Scaling group
Show Explanation

- Why C is correct: A CloudWatch alarm with a 1-minute period and 5 consecutive evaluation periods (datapoints to alarm = 5) means the alarm triggers after CPU exceeds 80% for 5 consecutive minutes. Linking the alarm action directly to an Auto Scaling policy provides automated capacity adjustment with zero custom code. - Why A is wrong: A 5-minute period with 1 evaluation period means the alarm triggers after a single 5-minute average exceeds the threshold. This only checks one data point, so a brief spike could trigger scaling, and it doesn't confirm the issue persists for 5 consecutive minutes. - Why B is wrong: Building a custom Lambda polling solution replicates what CloudWatch alarms and Auto Scaling policies already provide natively. This adds significant operational overhead (code maintenance, IAM permissions, error handling) for no benefit. - Why D is wrong: CloudTrail logs API calls, not performance metrics. It cannot detect high CPU utilization. CloudWatch is the correct service for metric-based monitoring.

Quiz 3 — Sample Question
A company runs a video transcoding application on EC2. The application reads large video files from one EBS volume and writes transcoded output to another. CloudWatch metrics show the output volume (a 500 GiB gp3 with default settings) consistently reaches its 125 MiB/s throughput limit, causing job delays. Which TWO actions will address the throughput bottleneck?
  • ✓ Increase the gp3 volume's provisioned throughput up to 1,000 MiB/s
  • Increase the gp3 volume's provisioned IOPS to 16,000
  • Migrate the volume to an io2 volume and provision maximum IOPS
  • Enable EBS Multi-Attach on the volume to distribute writes across instances
  • ✓ Use an EBS-optimized instance type that supports the required throughput
Show Explanation

- Why A is correct: gp3 volumes allow you to independently provision throughput up to 1,000 MiB/s, separate from volume size or IOPS. Since the bottleneck is the default 125 MiB/s throughput limit, increasing provisioned throughput directly resolves the issue. - Why E is correct: Even if the volume supports higher throughput, the EC2 instance must be EBS-optimized with sufficient EBS bandwidth to deliver that throughput. Both the volume and the instance must support the required throughput for the improvement to take effect. - Why B is wrong: IOPS (I/O operations per second) governs the number of read/write operations, not the data transfer rate (MiB/s). For large sequential file writes like video transcoding, throughput is the bottleneck, not IOPS. Increasing IOPS would not resolve the 125 MiB/s ceiling. - Why C is wrong: io2 volumes are designed for latency-sensitive, high-IOPS workloads such as databases. While io2 Block Express supports up to 4,000 MiB/s throughput, it is significantly more expensive than gp3 and over-provisioned for this use case. gp3 with increased throughput is the cost-effective solution. - Why D is wrong: EBS Multi-Attach is only available for io1 and io2 Provisioned IOPS volumes, not gp3. It is designed for clustered applications that need concurrent access from multiple instances in the same AZ, not for improving write throughput from a single instance.

Quiz 4 — Sample Question
A data engineering team has an AWS Lambda function that processes incoming CSV files from Amazon S3 and writes transformed records to Amazon DynamoDB. The function is triggered by S3 event notifications, and during peak hours hundreds of files arrive simultaneously. The team observes cold start latency of 3–5 seconds on each invocation, which delays downstream analytics dashboards. The team needs to eliminate cold starts while controlling costs. Which Lambda configuration meets these requirements?
  • ✓ Configure provisioned concurrency on the Lambda function to keep a specified number of execution environments pre-initialized and ready to respond instantly
  • Increase the function's reserved concurrency to 500 so that Lambda pre-warms 500 execution environments automatically
  • Increase the function's memory allocation to 10 GB, which automatically reduces cold start duration to near zero
  • Deploy the Lambda function inside a VPC with a NAT gateway to reduce initialization time by caching network connections
Show Explanation

- Why correct: Provisioned concurrency keeps a specified number of Lambda execution environments initialized, warm, and ready to serve requests with no cold start. When hundreds of files arrive during peak hours, the pre-initialized environments handle invocations immediately. You pay for provisioned concurrency whether invoked or not, but it guarantees sub-millisecond start times for the provisioned count. This directly addresses the 3–5 second cold start problem. - Why B is wrong: Reserved concurrency sets a maximum cap on concurrent executions for a function — it does NOT pre-warm environments. When a new invocation arrives and no warm environment exists, a cold start still occurs. Reserved concurrency limits the blast radius of a function but does nothing to eliminate cold starts. - Why C is wrong: Increasing memory allocation gives the function more CPU proportionally and can speed up execution time, but it does not eliminate cold starts. Cold start latency is driven by the runtime initialization (loading the deployment package, initializing the runtime, running init code), not by the amount of memory allocated. - Why D is wrong: Deploying a Lambda function inside a VPC actually increases cold start time because Lambda must attach an Elastic Network Interface (ENI) to the VPC during initialization. While Hyperplane improvements have reduced VPC cold starts significantly, VPC deployment does not eliminate cold starts and adds latency compared to non-VPC functions.

Quiz 5 — Sample Question
A SysOps administrator manages a web application behind an Application Load Balancer with an Auto Scaling group. During a recent flash sale, the application experienced degraded performance for several minutes before new instances launched. The administrator wants scaling to respond faster to sudden traffic spikes. Which TWO actions will reduce the time it takes for the Auto Scaling group to add capacity? (Choose TWO.)
  • ✓ Use a step scaling policy with a shorter CloudWatch alarm evaluation period
  • Increase the default cooldown period on the Auto Scaling group
  • Switch from target tracking to a simple scaling policy
  • Increase the health check grace period to 600 seconds
  • ✓ Configure a warm pool with pre-initialized instances
Show Explanation

- Why correct: Using a step scaling policy with shorter CloudWatch alarm evaluation periods (A) allows the Auto Scaling group to detect load changes faster and add capacity in graduated steps. Using a warm pool (E) keeps pre-initialized instances in a stopped or running state so they can be placed into service much faster than launching cold instances, which eliminates AMI boot time and application initialization delays. - Why A is correct: Step scaling policies react to alarm breaches with configurable step adjustments, and shortening the CloudWatch evaluation period (e.g., from 5 minutes to 1 minute) means the alarm triggers sooner. Combined, this significantly reduces reaction time. - Why B is wrong: Increasing the default cooldown period delays subsequent scaling actions, making the group slower to add more capacity — the opposite of what is needed. - Why C is wrong: Switching to a simple scaling policy is a downgrade. Simple scaling waits for the cooldown to expire before processing additional alarms, whereas step scaling can process alarms during cooldown and add capacity in proportion to the alarm breach size. - Why D is wrong: Increasing the health check grace period tells the Auto Scaling group to wait longer before checking instance health. This does not speed up scaling; it only delays when unhealthy instances are replaced. - Why E is correct: A warm pool pre-initializes instances so they are ready to serve traffic almost immediately when a scale-out event occurs, dramatically reducing launch latency.

Quiz 6 — Sample Question
A SysOps administrator notices that an Application Load Balancer (ALB) is routing traffic to only two of four registered EC2 instances. The other two instances show as "unhealthy" in the target group. The application is running and responding on port 443 on all four instances. What should the administrator check FIRST?
  • ✓ Verify that the target group health check is configured to use the correct port and path
  • Confirm the ALB is deployed in more than one Availability Zone
  • Upgrade the EC2 instances to a larger instance type
  • Enable ALB access logs to diagnose the issue
Show Explanation

- Why correct: When instances appear unhealthy in the target group despite the application running, the most common cause is a health check misconfiguration. The ALB health check may be probing the wrong port (e.g., port 80 when the app listens on 443) or the wrong path (e.g., / instead of /health). Verifying the health check port and path against the application's actual configuration is the correct first troubleshooting step. - Why A is wrong: This is the correct answer. - Why B is wrong: The ALB being in one AZ would affect which instances receive traffic, but it would not cause registered targets to show as unhealthy. Targets are marked unhealthy based on health check responses, not AZ placement. - Why C is wrong: Instance size affects performance under load, not health check status. If the application is running and responding, the instance type is not the issue. - Why D is wrong: ALB access logs record request details for auditing and debugging, but disabling them does not affect health check results or target health status.

Quiz 7 — Sample Question
A company runs critical workloads across Amazon EC2, Amazon RDS, and Amazon EBS. The operations team needs to automate daily backups for all these resources from a single location and enforce a 30-day retention policy. Which approach meets these requirements with the LEAST operational overhead?
  • ✓ Create an AWS Backup plan with a daily backup rule and a 30-day lifecycle policy targeting resources by tag
  • Write individual AWS CLI scripts scheduled via cron jobs on an EC2 instance to create snapshots for each service
  • Enable Amazon Data Lifecycle Manager (DLM) policies for EC2, RDS, and EBS separately
  • Use AWS Lambda functions triggered by Amazon EventBridge on a daily schedule to call snapshot APIs for each resource
Show Explanation

- Why A is correct: AWS Backup provides a centralized, fully managed service that can back up EC2, RDS, EBS, and many other AWS resources from a single backup plan. You define backup rules (frequency, retention) and assign resources by tags or resource ARNs. This eliminates the need to manage separate backup mechanisms for each service. - Why B is wrong: Writing custom CLI scripts on an EC2 instance introduces operational overhead — you must maintain the EC2 instance, manage cron reliability, handle script failures, and build retention logic yourself. AWS Backup handles all of this natively. - Why C is wrong: Amazon DLM supports EBS snapshots and EBS-backed AMIs, but it does not support RDS backups. You would still need a separate mechanism for RDS, which does not meet the 'single location' requirement. - Why D is wrong: Lambda functions with EventBridge can automate snapshots, but you must write and maintain code for each service's API, implement error handling, and build retention/cleanup logic. AWS Backup provides this out of the box with far less operational overhead.

Quiz 8 — Sample Question
A company runs an Auto Scaling group behind an Application Load Balancer. During a sudden traffic spike, the ASG launches new instances but the application experiences errors for several minutes because instances are added to the target group before the application is fully initialized. What should the SysOps administrator configure to resolve this issue?
  • Increase the ALB idle timeout to allow more time for instance startup
  • ✓ Configure a health check grace period on the Auto Scaling group that exceeds the application startup time
  • Set the ALB deregistration delay to match the application startup time
  • Add a scale-in cooldown period to the Auto Scaling group scaling policy
Show Explanation

- Why correct (B): The Auto Scaling health check grace period prevents the ASG from marking newly launched instances as unhealthy before the application has finished initializing. During this grace period, Auto Scaling does not act on health check failures from the ALB, giving the application time to start up and pass health checks. Setting this value to exceed the application's startup time ensures instances are not prematurely terminated and replaced. - Why A is wrong: The ALB idle timeout controls how long the load balancer keeps an idle connection open. It does not affect when instances are health-checked or when they start receiving traffic. - Why C is wrong: The deregistration delay controls how long the ALB waits before removing a deregistering target, which applies during scale-in (instance removal), not during instance launch. - Why D is wrong: A scale-in cooldown prevents rapid scale-in after a scaling activity, but the problem described is instances failing health checks during scale-out, not premature scale-in.

Quiz 9 — Sample Question
A SysOps administrator updates a production CloudFormation stack to change an EC2 instance type. The update fails and the stack enters UPDATE_ROLLBACK_FAILED state because the original instance type is no longer available in the Availability Zone. The administrator needs to bring the stack back to a working state. What should the administrator do?
  • Delete the stack and recreate it from scratch with the original template
  • ✓ Use the ContinueUpdateRollback action and skip the failed resource if necessary
  • Manually change the instance type in the EC2 console and wait for CloudFormation to detect the change
  • Create a new stack with a different name using the same template
Show Explanation

- Why correct (B): When a stack enters UPDATE_ROLLBACK_FAILED, the normal rollback has stalled because CloudFormation cannot restore one or more resources to their previous state. The ContinueUpdateRollback API action resumes the rollback. If specific resources cannot be rolled back (for example, the original instance type is unavailable), the administrator can skip those resources using the --resources-to-skip parameter, allowing the rest of the stack to complete the rollback. The skipped resources will be left in their current state and marked as UPDATE_COMPLETE. - Why A is wrong: Deleting and recreating the stack is destructive and unnecessary. It would cause downtime and data loss for any stateful resources (such as databases or EBS volumes) unless DeletionPolicy is set to Retain. ContinueUpdateRollback is the purpose-built mechanism for recovering from this state. - Why C is wrong: Manually modifying resources outside of CloudFormation creates drift between the stack template and the actual resource state. CloudFormation does not automatically detect or reconcile manual changes. The stack would remain in UPDATE_ROLLBACK_FAILED state regardless of manual changes. - Why D is wrong: Creating a new stack with a different name does not fix the broken stack. The original stack remains in UPDATE_ROLLBACK_FAILED state and must still be resolved. Additionally, resources with fixed physical names would conflict between the old and new stacks.

Quiz 10 — Sample Question
A SysOps administrator manages 200 Amazon EC2 instances across multiple AWS accounts. A critical security patch must be applied to all instances within 24 hours. The administrator needs to automate the patching without SSH access to each instance. Which approach meets this requirement with the LEAST operational overhead?
  • ✓ Use AWS Systems Manager Run Command with resource groups to execute the patch command across all target instances
  • Write a script that uses SSH to connect to each instance sequentially and apply the patch
  • Create a patched AMI and terminate all 200 instances so Auto Scaling replaces them
  • Open an AWS Support case requesting that AWS apply the patch to all instances
Show Explanation

- Why correct: AWS Systems Manager Run Command lets you remotely execute commands on managed instances at scale without SSH. You can target instances by tags or resource groups, apply rate controls, and results are logged centrally in Systems Manager. This is the purpose-built service for automating operational tasks on existing resources. - Why B is wrong: SSH scripting requires managing keys, maintaining bastion hosts, and does not scale well across accounts. It lacks centralized logging, error handling, and rate control that Systems Manager provides natively. - Why C is wrong: Terminating and replacing instances causes downtime, loses instance-specific state, and is not patching existing resources—it is reprovisioning. This is disruptive and operationally expensive. - Why D is wrong: AWS Support does not apply OS-level patches on customer EC2 instances. Under the shared responsibility model, OS patching is the customer's responsibility.

Quiz 11 — Sample Question
A company needs to produce hardened Amazon Linux 2 AMIs with a custom monitoring agent pre-installed. The AMIs must be available in four AWS Regions and rebuilt automatically whenever the base AMI is updated. The operations team wants minimal ongoing maintenance. What approach meets these requirements?
  • ✓ Create an EC2 Image Builder pipeline with a build component that installs the agent, a test component that validates the agent is running, and a distribution configuration targeting all required Regions
  • Manually launch an EC2 instance in each Region, install the agent, create an AMI, then copy the AMI to the remaining Regions using the console
  • Write a shell script on a single EC2 instance that installs the agent, snapshot the root volume, and share the snapshot cross-Region
  • Use AWS CloudFormation to launch an EC2 instance, install the agent via UserData, and create an AMI using a CloudFormation custom resource in every Region independently
Show Explanation

- Why correct: EC2 Image Builder provides a fully managed pipeline that automates AMI creation. Build components install software, test components validate the image, and distribution settings automatically copy the resulting AMI to multiple Regions. Image Builder can be triggered when the base AMI changes, satisfying the automated rebuild requirement with minimal maintenance. - Why B is wrong: Manually launching instances and copying AMIs in each Region is time-consuming, error-prone, and does not automate rebuilds when the base AMI updates — it requires ongoing manual effort. - Why C is wrong: Snapshots are not AMIs. A snapshot of a root volume must still be registered as an AMI before it can launch instances. Quiz 12 — Sample Question
A company's compliance team requires that all IAM users have passwords with a minimum length of 14 characters, must include uppercase and lowercase letters, numbers, and symbols, and must be rotated every 90 days. A SysOps administrator needs to enforce these requirements across the AWS account. Which TWO actions should the administrator take in the IAM password policy?
  • ✓ Enforce minimum length and complexity requirements (uppercase, lowercase, numbers, symbols)
  • Require all users to use access keys instead of passwords
  • ✓ Set maximum password age to 90 days to enforce periodic rotation
  • Use AWS Organizations SCP to block weak passwords
  • Enable MFA to replace password requirements
Show Explanation

- Why correct: An IAM password policy allows administrators to enforce complexity requirements (minimum length, character types) and rotation intervals (maximum password age). These are both configurable settings within the IAM account password policy and directly address the compliance requirement. - Why A (Enforce minimum length and complexity) is correct: The IAM password policy supports setting minimum length (6–128 characters) and requiring uppercase, lowercase, numbers, and symbols. - Why B (Require all users to use access keys instead of passwords) is wrong: Replacing passwords with access keys does not address password compliance — it sidesteps the requirement rather than enforcing it. Access keys are for programmatic access, not console login. - Why C (Set maximum password age for rotation) is correct: The IAM password policy supports setting a maximum password age (1–1095 days), which enforces periodic rotation as required by the compliance mandate. - Why D (Use AWS Organizations SCP to block weak passwords) is wrong: SCPs control which AWS API actions are allowed or denied — they cannot enforce IAM password complexity or rotation rules. Password policy is an IAM account-level setting. - Why E (Enable MFA to replace password requirements) is wrong: MFA adds a second authentication factor but does not enforce password complexity or rotation. Both controls serve different purposes and MFA does not substitute for a password policy.

Quiz 13 — Sample Question
A company stores sensitive financial records in Amazon S3. The security team requires that the data be encrypted using keys that the company fully manages, including the ability to rotate keys on a custom schedule and to disable or delete keys immediately if compromised. Which encryption approach meets these requirements?
  • SSE-S3 with Amazon S3 managed keys
  • ✓ SSE-KMS with a customer managed key (CMK) in AWS KMS
  • Client-side encryption with keys stored in an EC2 instance environment variable
  • SSE-S3 with automatic key rotation disabled
Show Explanation

- Why correct: SSE-KMS with a customer managed key (CMK) gives the company full control over the key lifecycle — including custom rotation schedules, the ability to disable or schedule deletion of keys, and detailed usage auditing via CloudTrail. This meets the requirement for company-managed keys with immediate revocation capability. - Why A is wrong: SSE-S3 uses keys that AWS fully manages on your behalf. You cannot control rotation schedules, disable keys, or delete them — AWS handles everything transparently, so the company lacks the required key management control. - Why C is wrong: Storing encryption keys in EC2 environment variables is insecure — they can be exposed through instance metadata, process listings, or logs. You cannot disable rotation or manage S3-managed keys — they are opaque to the customer, so this option does not satisfy the requirement for company-managed keys.

Quiz 14 — Sample Question
A company uses AWS Organizations with multiple member accounts. The networking team manages a central networking account and needs application accounts to deploy resources into centrally managed VPC subnets without allowing those teams to modify the subnet configuration. Which approach should the SysOps administrator implement?
  • ✓ Use AWS RAM to share VPC subnets; participant accounts can launch resources into shared subnets but cannot modify subnet attributes
  • Use VPC peering between accounts and replicate the same subnet CIDRs in each application account
  • Create identical subnets in each account using CloudFormation StackSets so each team manages its own
  • Grant each application account an IAM cross-account role with full VPC permissions in the central networking account
Show Explanation

- Why correct: AWS RAM enables sharing VPC subnets across accounts within an Organization. Participant accounts can launch resources (EC2 instances, RDS databases, Lambda functions, etc.) into shared subnets, but only the owner account can modify or delete the subnet itself. This enforces centralized network control while enabling application teams to deploy resources. - Why B is wrong: VPC peering connects two VPCs for routing traffic between them, but it does not share subnets. Each account would still own and manage its own subnets independently, defeating the goal of centralized network management. - Why C is wrong: StackSets can deploy identical subnets in each account, but each account would independently own its subnets. This means application teams could modify or delete their subnets, removing centralized control and increasing IP address waste through duplication. - Why D is wrong: Granting full VPC permissions in the central networking account violates the principle of least privilege. Application teams could modify shared networking infrastructure (route tables, NACLs, etc.), creating security and operational risks.

Quiz 15 — Sample Question
A SysOps administrator needs to configure a VPC so that EC2 instances in a public subnet can communicate with the internet. The instances must be reachable from the internet and must also initiate outbound connections. What combination of steps is required?
  • ✓ Attach an internet gateway to the VPC, assign public IP addresses to the instances, and add a route to 0.0.0.0/0 pointing to the internet gateway in the subnet's route table
  • Create a NAT gateway in the subnet and route all traffic through it
  • Attach an internet gateway to the VPC only; no other changes are needed
  • Assign Elastic IP addresses to the instances without attaching an internet gateway
Show Explanation

- Why correct: For EC2 instances in a public subnet to reach the internet, three things are required: (1) an internet gateway attached to the VPC, (2) a route in the subnet's route table sending 0.0.0.0/0 traffic to the internet gateway, and (3) instances must have a public IPv4 address (either auto-assigned or an Elastic IP). All three are necessary. - Why B is wrong: A NAT gateway is used for private subnets to allow outbound-only internet access. Instances behind a NAT gateway cannot receive inbound connections from the internet, which does not meet the requirement for a public-facing subnet. - Why C is wrong: Attaching an internet gateway alone is not sufficient. Without a route table entry directing 0.0.0.0/0 to the internet gateway and without public IPs on the instances, traffic will not flow to the internet. - Why D is wrong: Elastic IP addresses are public IPs, but without an internet gateway attached to the VPC, there is no path for traffic to reach the internet. The internet gateway is the component that enables communication between the VPC and the internet.

Quiz 16 — Sample Question
A company runs a web application with users in the US and Europe. Each Region has its own deployment behind an Application Load Balancer. The operations engineer wants to route each user to the deployment that provides the fastest response time. Which Route 53 routing policy should the engineer configure?
  • Simple routing policy
  • Weighted routing policy
  • ✓ Latency-based routing policy
  • Multivalue answer routing policy
Show Explanation

- Why correct: Latency-based routing evaluates latency between the user and each AWS Region, then returns the record for the Region with the lowest latency. This is the purpose-built policy for directing users to the fastest endpoint. - Why A is wrong: Simple routing returns a single record (or multiple in random order) with no intelligence about latency or health. It cannot direct users to the closest Region. - Why B is wrong: Weighted routing distributes traffic by percentage (e.g., 80/20). It does not measure or optimize for latency; it is used for load distribution or canary deployments. - Why D is wrong: Multivalue answer routing returns multiple healthy IP addresses and lets the client pick one randomly. It provides basic health-check filtering but does not optimize for latency.

Quiz 17 — Sample Question
An operations engineer notices that EC2 instances in a private subnet cannot download software updates from the internet, even though a NAT gateway exists in a public subnet. The VPC has an internet gateway attached. What is the MOST likely cause?
  • The internet gateway is not attached to the private subnet's route table
  • ✓ The private subnet's route table is missing a 0.0.0.0/0 route pointing to the NAT gateway
  • The security group is blocking return traffic from the internet
  • The VPC peering connection needs to be configured for internet access
Show Explanation

- Why correct: When an instance in a private subnet cannot reach the internet, the most common cause is a missing or misconfigured route to a NAT gateway. The route table associated with the private subnet must contain a route for 0.0.0.0/0 pointing to a NAT gateway in a public subnet. Without this route, outbound traffic has no path to the internet. - Why A is wrong: Internet gateways (IGWs) are attached at the VPC level and provide internet access for public subnets. Adding a route to the IGW from a private subnet would expose instances directly to the internet, which defeats the purpose of a private subnet and is not the correct approach. - Why C is wrong: Security groups are stateful — if outbound traffic is allowed, the return traffic is automatically permitted. Security groups do not block return traffic for established connections, so this would not cause the described issue. - Why D is wrong: While VPC peering enables communication between VPCs, it does not provide internet access. VPC peering is used for private connectivity between VPCs, not for routing traffic to the internet.

Quiz 18 — Sample Question
A company launches an EC2 instance in a public subnet of a VPC. The instance has a public IP address and the subnet's route table has a 0.0.0.0/0 route to the internet gateway. However, the instance cannot reach the internet. The security group allows all outbound traffic. What is the MOST likely cause?
  • The instance needs an Elastic IP address instead of an auto-assigned public IP address
  • The subnet's route table is missing the local route for the VPC CIDR block
  • ✓ The network ACL associated with the subnet is denying outbound traffic or the return inbound traffic
  • The internet gateway has not been attached to the VPC
Show Explanation

- Why correct: Network ACLs are stateless and evaluated separately for inbound and outbound traffic. Even though the security group allows all outbound traffic, the NACL must also allow both the outbound request AND the inbound return traffic (on ephemeral ports). The default NACL allows all traffic, but if a custom NACL has been applied with restrictive rules — for example, missing an inbound allow rule for return traffic on ephemeral ports (1024-65535) — connectivity will fail. Since the question confirms the security group, route table, public IP, and IGW route are all correct, the NACL is the most likely remaining cause. - Why A is wrong: Both auto-assigned public IP addresses and Elastic IP addresses enable internet connectivity through an internet gateway. An Elastic IP provides a static address that persists across instance stop/start cycles, but either type works for internet access. The choice between them does not affect basic connectivity. - Why B is wrong: The local route for the VPC CIDR block is automatically present in every route table and cannot be removed or modified. It is always there by default, so it cannot be the cause of the issue. - Why D is wrong: If the internet gateway were not attached to the VPC, the 0.0.0.0/0 route pointing to it would show a 'blackhole' status in the route table. The question states the route table has a route to the IGW, implying the IGW is attached and the route is active.

Quiz 19 — Sample Question
A SysOps administrator notices that an Auto Scaling group is not launching new instances during a traffic spike, even though the scaling policy has been triggered. The launch template references an AMI that was created in a different AWS Region. What is the MOST LIKELY cause of the issue?
  • The Auto Scaling group has reached its maximum capacity limit
  • ✓ The AMI referenced in the launch template is not available in the current Region
  • The scaling policy cooldown period has not yet expired
  • The Elastic Load Balancer health check is failing for new instances
Show Explanation
